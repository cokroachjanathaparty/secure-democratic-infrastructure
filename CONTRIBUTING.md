
# Contributing to Secure Democratic Infrastructure (SDI)

Thank you for your interest in contributing to Secure Democratic Infrastructure (SDI).

SDI is an open constitutional technology initiative focused on building transparent, secure, auditable, and privacy-preserving democratic infrastructure for large-scale national elections.

We welcome contributions from:

- Security researchers
- Cryptographers
- Backend engineers
- Infrastructure engineers
- Blockchain developers
- Civic technologists
- Academics
- Open-source contributors
- Governance researchers
- Accessibility specialists

---

# Core Contribution Principles

All contributions must align with the following principles:

- Security-first engineering
- Constitutional decentralization
- Transparency and auditability
- Privacy preservation
- Open-source collaboration
- Reproducible systems
- Responsible disclosure
- Research-grade quality standards

# Development Standards

## Mandatory Requirements

All contributions must follow:

- Conventional commits
- Semantic versioning
- Infrastructure-as-code
- OpenAPI-first APIs
- Mandatory peer review
- Mandatory security review
- Reproducible builds where applicable

---

# Branch Naming Standards

Examples:

```text
feature/validator-consensus
fix/identity-verification
security/firmware-attestation
docs/api-specification
research/zk-proof-analysis
````

---

# Commit Standards

Use conventional commit format:

```text
feat: add validator synchronization module
fix: resolve encrypted vote queue issue
docs: update governance documentation
security: improve TPM attestation validation
```

---

# Pull Request Process

## Before Opening a PR

Ensure:

* Code builds successfully
* Tests pass
* Documentation is updated
* Security implications are documented
* Threat models are updated if required

## Pull Request Requirements

Each PR should include:

* Clear summary
* Technical rationale
* Security considerations
* Testing evidence
* Related issue references

---

# Security Review Requirements

The following changes require mandatory security review:

* Cryptographic logic
* Authentication systems
* Identity verification
* Blockchain consensus
* Validator infrastructure
* Firmware systems
* Key management
* Secure enclave integrations

---

# Code Review Standards

All contributions are reviewed for:

* Security
* Maintainability
* Performance
* Reliability
* Auditability
* Documentation quality
* Architectural consistency

---

# Prohibited Contributions

The following are not accepted:

* Hardcoded secrets
* Obfuscated code
* Unsigned critical binaries
* Unreviewed cryptographic implementations
* Closed-source dependencies without review
* Code violating privacy guarantees

---

# Responsible Disclosure

Do not publicly disclose security vulnerabilities.

Please follow the procedures defined in:

`SECURITY.md`

before reporting vulnerabilities.
